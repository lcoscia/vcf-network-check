Basic Configuration

Min 4 for VCF bring-up (vSAN FTT=1 + HA)
Déploiement standard sur un seul site. Tous les hôtes et VLANs sont locaux.
vSAN Stretched Cluster : Topologie VCF 9 standard pour les déploiements bi-site avec vSAN comme stockage primaire. Les données sont répliquées synchrone entre les 2 sites. Un hôte Witness tiers assure le quorum.

⚡ Prérequis réseau (Broadcom TechDocs VCF 9)
  • Latence inter-sites (data ↔ data) : ≤ 5 ms RTT
  • Latence data ↔ Witness : ≤ 200 ms RTT (100 ms one-way recommandé) ; si > 10 hôtes/site : < 100 ms RTT
  • Bande passante inter-sites : 10 Gbps minimum (ESA et OSA)
  • Witness : 2 Mbps par tranche de 1 000 composants vSAN
🖧 VLANs et L2/L3
  • vSAN VMkernel : L2 obligatoire entre les 2 sites (même VLAN ID)
  • VM Management Network : L2 étendu sur les 2 sites — même VLAN ID requis
  • ESXi Management (vmk0) : L2 recommandé pour l'automatisation VCF Ops (ajout de serveur)
  • vMotion : L2 recommandé pour conserver les automatismes VCF Ops
  • NSX Host TEP : L3 accepté (BGP/OSPF), L2 recommandé pour simplifier
  • NFS (si stockage NFS) : L2 obligatoire sur les 2 sites
✅ Prérequis architecture
  • Nombre d'hôtes identique sur chaque site (symétrie obligatoire)
  • vSAN Witness Host requis sur un 3ème site indépendant — ne fait pas tourner de VMs, même version ESXi que les data hosts
  • Dans VCF : le cluster Management Domain doit être étendu en premier avant tout Workload Domain
  • Hôtes DPU (SmartNIC) et hôtes sur subnets mixtes : non supportés en stretched
  • Mapping vmnic → vDS identique entre les 2 zones
  • SMP-FT non supporté avec Site Disaster Tolerance ≥ 1
  • vSAN ESA et OSA sont tous deux supportés en stretched
Sources : Broadcom — Bandwidth & Latency · vSAN Stretched Network Requirements
Stretched (vMSC) : Le cluster est étendu sur 2 sites actifs-actifs avec vSAN Stretched Cluster.

⚡ Prérequis réseau (source : Broadcom TechDocs VCF 9)
  • Latence inter-sites (data ↔ data) : ≤ 5 ms RTT
  • Latence data ↔ Witness : ≤ 200 ms RTT (100 ms one-way recommandé) ; si > 10 hôtes/site : < 100 ms RTT
  • Bande passante inter-sites : 10 Gbps minimum (ESA et OSA)
  • Witness : 2 Mbps par tranche de 1 000 composants vSAN
🖧 Prérequis réseau VLANs
  • VM Network (Management VMs) : L2 étendu sur les 2 sites — même VLAN ID requis
  • ESXi Management (vmk0) : peut être L3 (zone-specific), mais L2 recommandé pour l'automatisation VCF Ops (ajout de serveur)
  • vMotion : L2 recommandé pour conserver les automatismes VCF Ops ; L3 possible mais perte d'automatisation
  • vSAN : L2 obligatoire entre les 2 sites (même VLAN ID)
  • NFS (si stockage NFS) : L2 obligatoire entre ESXi et serveur NFS sur les 2 sites
  • NSX Host TEP : L3 accepté (BGP/OSPF), L2 recommandé pour simplifier la configuration
✅ Prérequis architecture vSAN Stretched
  • Nombre d'hôtes identique sur chaque site (symétrie obligatoire)
  • vSAN Witness Host requis sur un 3ème site (ou vSAN Witness Appliance) — ne fait pas tourner de VMs, même version ESXi que les data hosts
  • Dans VCF : le cluster Management Domain doit être étendu en premier avant tout Workload Domain
  • Hôtes DPU (SmartNIC) et hôtes sur subnets mixtes : non supportés en stretched
  • Mapping vmnic → vDS identique entre les 2 zones
  • SMP-FT non supporté avec Site Disaster Tolerance ≥ 1
Sources : Broadcom TechDocs — Bandwidth & Latency · vSAN Stretched Network Requirements
Partially Stretched : Les composants de management restent sur le site principal. Seuls des hôtes compute sont étendus vers un site secondaire.
  • Les VLANs de management ne nécessitent pas d'extension L2 complète — seules les VLANs de la couche compute (ESXi Mgmt secondaire, vMotion compute) doivent être étendues.
  • Le site secondaire héberge uniquement des VMs de workload, pas les composants VCF.
  • Différent du mode Stretched complet : SDDC Manager, vCenter, NSX Managers restent au site principal.
Min 2 recommandé (TEP HA actif-actif)
NFS : VLAN NFS généré à la place de vSAN. L2 obligatoire entre hôtes ESXi et serveur NFS. VMFS : datastores sur baie SAN (iSCSI ou FC). Aucun VLAN de stockage VMkernel généré (iSCSI : à configurer manuellement via Additional Services si VLAN dédié requis).
L2 Adjacency Confirmed : Confirme que les VLANs requis pour le bring-up VCF sont accessibles en L2 depuis le SDDC Manager / Cloud Builder :
  • ESXi Management (vmk0 des hôtes)
  • Management VM Network (SDDC Manager, vCenter, NSX Managers)
  • vMotion (requis par le bring-up pour la validation HA)
Ce n'est pas une exigence permanente (le NSX TEP et les uplinks Edge peuvent être L3 en production), mais le bring-up Cloud Builder valide la connectivité L2 initiale.

NSX Configuration

Fleet & Platform Services Network Model

Modèle 1 — Shared Management VM Network : Fleet et tous les services platform partagent le Management VM Network (port group existant). Déploiement Day 0. Aucun VLAN supplémentaire requis. Simple, recommandé pour les petits déploiements.
Modèle 2 — Dedicated Fleet VLAN : Un VLAN dédié est créé pour le Fleet Appliance et les nœuds des services platform (Ops, Logs, Networks, Automation). Les Collectors restent toujours dans le Management VM Network.
  • Configuration Day 2 (post bring-up via API)
  • Un VLAN physique supplémentaire est requis et généré dans VLAN Design
  • Les collecteurs (Remote Collectors, Proxy VMs) restent dans Management VM Network
Modèle 3 — NSX VLAN Segment : Segment NSX backé par un VLAN physique (VLAN transport zone). Le routage est assuré par le fabric physique.
  • Créer une VLAN transport zone dans le NSX Manager du Management Domain
  • Créer un segment NSX sur cette nouvelle transport zone
  • Attacher la VLAN transport zone au transport node profile du cluster primary
  • NSX Edge non requis — routage physique
  • Pas de NSX Federation ni de stretching overlay possible
Source : Broadcom TechDocs — NSX VLAN Segment
Modèle 4 — NSX Overlay Segment : Segment NSX overlay (Geneve). Isolation réseau complète, policies de sécurité dédiées.
  • NSX Edge Cluster obligatoire — avec Tier-0 et Tier-1 gateways configurés
  • BGP requis avec peer ASN et passwords pour la connectivité externe
  • NSX Load Balancer disponible pour les composants Fleet
  • Supporte NSX Federation pour le stretching multi-sites (DR)
  • Les VCF Operations Collectors sont toujours déployés dans le Management VM Network
  • Pas de VLAN ID physique — segment overlay (encapsulation Geneve)
Source : Broadcom TechDocs — NSX Overlay Segment
⚠️ NSX Edge non déployé — Le modèle NSX Overlay Segment nécessite un NSX Edge Cluster. Activez "NSX Edge Deployed" dans la section NSX Configuration ci-dessus.

Purpose

The VCF 9 Network Planner automates the pre-deployment network design process for VMware Cloud Foundation 9 deployments. It replaces manual spreadsheet workflows with a single, self-consistent planning document.

From a handful of configuration choices, the tool derives the complete set of VLANs, appliance IP allocations, VIPs, and runs architectural validation — ready for export to Excel or JSON.
