5 · Liste des Anomalies — Par Sévérité

⛔ Critique C-01 · NSX TEP IPs sous-estimées — MD, WD1 et WD2
Le plan déclare 6 IPs TEP par domaine. Le calcul VCF 9 Planning & Preparation Workbook exige :
  • 4 hôtes ESXi × 2 TEP (ECMP) = 8 IPs par domaine
  • 2 NSX Edge nodes × 2 TEP = 4 IPs par domaine
  • Total = 12 IPs minimum par domaine · 36 IPs TEP au total pour MD + WD1 + WD2
Les subnets /24 déclarés sont largement suffisants en capacité. C'est uniquement le comptage déclaré (6) qui est erroné et doit être corrigé dans votre plan avant saisie dans le VCF 9 Preparation Workbook.
Action : Corriger le comptage à 12 IPs TEP par domaine dans votre plan d'adressage. Ajuster les subnets à /27 (30 usables) si vous souhaitez optimiser l'espace IP (option recommandée pour rationaliser le plan).
⚠️ Majeure M-01 · Gateways non définies pour aucun des 20 réseaux
Le VCF 9 Planning & Preparation Workbook exige une gateway (SVI) par réseau. Sans gateway documentée, la validation pre-flight du SDDC Manager échouera sur le test de connectivité réseau. Ces SVIs doivent exister sur vos switches ToR avant le déploiement.
Action : Définir une gateway pour chaque sous-réseau (convention suggérée : .1 — ex. 10.10.10.1, 10.10.20.1…). Valider que les SVIs correspondantes sont créées sur vos ToR avant de lancer le déploiement SDDC Manager.
⚠️ Majeure M-02 · NSX Manager Cluster — IPs nœuds 2 & 3 et VIPs non allouées
En VCF 9, le NSX Manager est toujours déployé en cluster de 3 nœuds (obligatoire pour la production). Le plan actuel ne documente qu'1 IP NSX Manager par domaine. Il manque donc, pour chacun des 3 domaines :
  • NSX Manager Nœud 2 : 1 IP (×3 domaines = 3 IPs)
  • NSX Manager Nœud 3 : 1 IP (×3 domaines = 3 IPs)
  • NSX Manager VIP Cluster : 1 IP (×3 domaines = 3 IPs)
Total à allouer : 9 IPs supplémentaires dans le réseau MD VM Management.
Action : Allouer 9 IPs supplémentaires dans 10.10.10.0/xx pour les nœuds 2 & 3 et VIPs des clusters NSX Manager MD, WD1 et WD2. Voir tableau Section 3 pour les IPs proposées.
ℹ️ Mineure m-01 · Subnets /24 oversized — 14 réseaux sur 20 concernés
La majorité des réseaux ESXi Management, vMotion, vSAN et NSX Edge Uplinks utilisent des /24 (254 usables) pour seulement 2 à 4 IPs requises. Ce surdimensionnement n'est pas bloquant si l'espace IP est disponible, mais impacte l'efficacité du plan d'adressage et peut compliquer la numérotation VLAN future.
Action : Adopter les subnets recommandés du tableau Section 2 (/28 pour ESXi Mgmt/vMotion/vSAN, /29 pour Edge Uplinks, /27 pour NSX TEP). Si l'espace IP n'est pas contraint, les /24 actuels restent techniquement valides.
ℹ️ Mineure m-02 · Fleet Management — IPs composants non détaillées individuellement
Le réseau Fleet Management (10.10.80.0/24) déclare 6 IPs sans détail par composant (VCF Ops, VCFA, Aria Logs, Aria Networks). Les clusters de ces composants nécessitent 14 IPs au total. Une documentation IP par composant est requise pour la validation SDDC Manager.
Action : Documenter l'allocation IP composant par composant dans le Fleet Management. Exemple : VCF Operations Nœud 1 = .10, Nœud 2 = .11, Nœud 3 = .12, VIP = .13 · VCF Automation : .20–.23 · etc.
ℹ️ Info I-01 · VLAN IDs — À définir en Phase 2 (non bloquant à ce stade)
Les VLAN IDs seront assignés après validation des prérequis IP, selon votre plan VLAN physique (ToR). Cette démarche est correcte et recommandée (IP d'abord, VLAN ensuite). Ils devront être communiqués au moment de la saisie dans le VCF 9 Planning & Preparation Workbook.
À prévoir : 20 VLAN IDs distincts (plage 1–4094), sans doublons inter-domaines ni conflits avec l'infrastructure physique existante. Convention suggérée : MD = 100–107, WD1 = 110–118, WD2 = 120–125.
